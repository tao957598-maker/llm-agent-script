## 第一部分：AI执行洞察

### 1. 高优先级问题 (highPriorityIssue)
识别当前最紧急、最需要关注的质量风险点。考虑：
- 如果失败率高(>30%)，强调严重质量问题
- 如果有特定模块失败集中，指出该模块

- 字数控制在100-200字

### 2. 疑似不稳定用例 (unstableCases)
分析可能的Flaky Test（不稳定测试）。考虑：
- 如果某些模块通过率波动大，指出这些模块
- 如果失败原因多为"元素未找到"或"超时"，可能是环境问题导致的不稳定
- 如果通过率在70%-90%之间，可能存在边界条件问题
- 如果没有明显不稳定迹象，给出肯定评价
- 字数控制在100-200字

### 3. 执行优化建议 (optimizationSuggestion)
提供可操作的改进建议。考虑：
- Agent成功率低于90%时，建议检查环境和配置
- 失败率高时，建议优先修复缺陷
- 覆盖率低时，建议补充测试用例
- 如果有特定失败类型集中，建议针对性优化
- 如果整体表现良好，给予肯定并建议保持
- 字数控制在100-200字

## 第二部分：最终结论

### 1. 质量评估 (qualityAssessment)
对整体质量状况进行评估。考虑：
- 通过率>=90%：质量风险可控，满足发布标准
- 通过率70%-90%：存在一定质量风险，需要关注
- 通过率<70%：质量风险较高，不建议发布
- 结合覆盖率、失败分布、稳定性评分等因素综合评估
- 字数控制在80-150字

### 2. 高风险模块 (highRiskModule)
识别并说明高风险模块。考虑：
- 如果有模块失败率特别高，明确指出

- 如果没有明显高风险模块，说明"未发现高风险模块"
- 可以引用具体的失败原因分布
- 字数控制在50-100字

### 3. 修复建议 (recommendation)
提供清晰、可执行的修复建议。考虑：
- 优先级的排序（先修什么，后修什么）
- 是否需要补充测试
- 是否需要重新测试
- 是否可以直接发布
- 分条列出，条理清晰
- 字数控制在80-150字
