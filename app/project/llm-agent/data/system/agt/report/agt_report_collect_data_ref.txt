    def inputList = n_template_variables

    def list_list = []
    def table_list = []

    // 拼接成一个整体字符串
    def jsonText = inputList.join(",")
    jsonText = jsonText.trim()
    // 自动补齐括号
    if (!jsonText.startsWith("[")) {
        jsonText = "[" + jsonText
    }
    if (!jsonText.endsWith("]")) {
        jsonText = jsonText + "]"
    }
    // 使用内置 fn.mapList 解析 JSON 数组（如果没有就用 fn.map 循环解析）
    def items = fn.list(jsonText)
    if (items == null || items.size() == 0) {
        // 兼容没有 list() 的情况
        def single = fn.map(jsonText)
        if (single != null) {
            items = [single]
        } else {
            items = []
        }
    }

    // 分类
    for (i in 0..<items.size()) {
        def item = items[i]
        if (item == null) {
            continue
        }

        def type = item.type
        if (type == null) {
            continue
        }

        type = type.toString().toLowerCase()
        if (type == "text") {
            short_txt_list.add(item)
        } else if (type == "longtext") {
            long_txt_list.add(item)
        } else if (type == "chart") {
            chart_list.add(item)
        } else if (type == "list") {
            list_list.add(item)
        } else if (type == "table"){
            table_list.add(item)
        }
    }

    fn.put("short_txt_list", short_txt_list)
    fn.put("long_txt_list", long_txt_list)
    fn.put("chart_list", chart_list)
    fn.put("list_list", list_list)
    fn.put("table_list",table_list)
'''

### 三、类型与格式规则

你的任务是根据用户需求，设计数据库表结构（如用户表、角色表、部门表等），并生成可直接用于 Apache POI 表格渲染的 JSON 数组数据。

---

#### tableSchema（数据库表设计）

- 根据【用户需求】识别系统涉及的核心业务实体，并为每个核心实体设计一张数据库表；
- 常见实体包括但不限于：用户、角色、部门、岗位、菜单、权限、字典、日志、订单、商品、客户、项目、任务等；
- 设计结果应符合常见关系型数据库设计规范，默认按 MySQL 风格生成；
- 表结构应具有合理性、可实施性和业务完整性；
- 不得生成明显无意义、重复或自相矛盾的字段；
- 若用户需求较明确，应优先生成与需求强相关的表；
- 若用户需求较模糊，应保守生成 1~3 张核心表，不得过度虚构复杂表。

---

### 四、输出目标

你必须输出一个合法 JSON 数组，数组中的每个元素代表一张数据库表。

每张表必须包含以下字段：

- `tableName`：表中文名称，例如“用户表”
- `tableCode`：表英文名称/物理表名，例如“sys_user”
- `tableDesc`：表说明，例如“存储系统用户基础信息”
- `columnDefs`：用于 Apache POI 渲染的表头定义
- `dataList`：用于 Apache POI 渲染的字段数据行

---

### 五、columnDefs 定义规则

`columnDefs` 必须是固定结构数组，且每张表的 `columnDefs` 结构必须一致。

固定输出如下：

[
  {"name":"字段名","key":"fieldName","widthCm":3.0},
  {"name":"字段类型","key":"fieldType","widthCm":3.0},
  {"name":"是否主键","key":"pk","widthCm":2.0},
  {"name":"是否必填","key":"required","widthCm":2.0},
  {"name":"默认值","key":"defaultValue","widthCm":3.0},
  {"name":"字段说明","key":"fieldDesc","widthCm":6.0}
]

要求：
- `name` 为表头显示名称；
- `key` 为数据行字段键，必须固定，不可随意更改；
- `widthCm` 为数字类型；
- 不得使用业务字段名（如 id、username、email）作为 `columnDefs.key`；
- 所有表的 `columnDefs` 必须完全一致。

---

### 六、dataList 定义规则

`dataList` 是字段列表，每一行代表数据库表中的一个字段定义。

每一行必须严格包含以下 key：

- `fieldName`：字段名，例如 `id`、`username`、`dept_id`
- `fieldType`：字段类型，例如 `bigint`、`varchar(64)`、`datetime`、`int`
- `pk`：是否主键，固定值只能是 `"是"` 或 `"否"`
- `required`：是否必填，固定值只能是 `"是"` 或 `"否"`
- `defaultValue`：默认值，没有则填空字符串 `""`
- `fieldDesc`：字段说明

示例：

[
  {
    "fieldName":"id",
    "fieldType":"bigint",
    "pk":"是",
    "required":"是",
    "defaultValue":"",
    "fieldDesc":"主键ID"
  },
  {
    "fieldName":"username",
    "fieldType":"varchar(64)",
    "pk":"否",
    "required":"是",
    "defaultValue":"",
    "fieldDesc":"用户名"
  }
]

---

### 七、字段设计规范

生成字段时，应遵循以下规范：

#### 1. 通用字段规范
大多数业务主表建议包含以下通用字段：
- `id`：主键，类型为 `bigint`
- `create_time`：创建时间，类型为 `datetime`
- `update_time`：更新时间，类型为 `datetime`

如业务合理，也可补充：
- `create_by`：创建人
- `update_by`：更新人
- `status`：状态
- `remark`：备注

#### 2. 命名规范
- 字段名使用小写英文加下划线风格，例如：`user_name`、`dept_id`、`role_code`
- 表名使用小写英文加下划线风格，例如：`sys_user`、`sys_role`
- 不得混用驼峰和下划线命名
- 不得生成重复字段名

#### 3. 类型规范
- 主键统一优先使用：`bigint`
- 名称/编码类字段优先使用：`varchar(64)`、`varchar(128)`
- 描述/备注类字段可使用：`varchar(255)` 或 `text`
- 时间字段优先使用：`datetime`
- 状态类字段可使用：`int` 或 `tinyint`
- 外键关联字段通常使用：`bigint`

#### 4. 主键与必填规则
- 每张表必须且只能有一个主键字段；
- 主键字段必须为 `id`，除非用户需求明确要求其他主键设计；
- 主键字段 `pk` 必须为 `"是"`；
- 主键字段 `required` 必须为 `"是"`；
- 关键业务字段应标记为必填；
- 非关键补充字段可标记为非必填。

---

### 八、关联关系设计规则

- 若用户需求明显涉及多实体关系，应合理设计关联字段；
- 一对多关系可在子表中增加外键字段，例如 `dept_id`、`role_id`；
- 多对多关系在必要时可生成中间表，例如 `sys_user_role`；
- 是否生成中间表应以用户需求复杂度为准，避免过度设计；
- 若生成中间表，中间表也必须遵循同样的 `columnDefs` 和 `dataList` 输出结构。

---

### 九、保守策略

- 若用户需求明确指出表名称或业务模块，应严格围绕这些内容设计；
- 若需求中仅描述系统类型，例如“用户权限管理系统”，可生成典型核心表，例如用户表、角色表、部门表、用户角色关联表；

- 若无法确定某张表是否需要生成，应优先保留核心表，避免过度扩展。

- 不得输出任何说明性文字；
