define('tpl_page_summary_api', '''
You are extracting structured API requirement items from one page slice of contract material such as REST, OpenAPI, Swagger, interface spec, request/response schema, or error code documentation.

Return JSON array only.

Hard rules:
1. Return one JSON array only. No markdown, no explanation, no comments.
2. `requirement_type` must use one fixed English enum only: `Background`, `Functional`, `Compliance`, `Other`.
3. All user-visible text fields other than fixed enums must be written in {{lang_output_text}}.
4. If this page has no clear API-contract content, return [].
5. Do not invent endpoints, methods, parameters, status codes, headers, or schema rules.
6. Ignore TOC, navigation, task metadata, project admin text, or non-contract chatter.
7. Split distinct operations when the page clearly describes different testable API surfaces.

Output schema:
```json
[{
  "page": 1,
  "requirement_type": "Functional",
  "section": "API area or tag",
  "title": "METHOD path or operation title",
  "description": "Concise API purpose and key constraints",
  "business_rules": ["Rule 1"],
  "acceptance_criteria": ["Criterion 1"],
  "dependencies": [],
  "testability": true
}]
```

Field guidance:
- `section`: resource group, API area, or service area.
- `title`: prefer `METHOD path` when available.
- `description`: concise contract meaning, auth mode, critical params, or main behavior.
- `business_rules`: required headers, auth rules, enum constraints, schema restrictions, rate limit, deprecation, error mapping.
- `acceptance_criteria`: directly testable contract expectations such as status code and response rules.
- `testability`: false only when the page is glossary/background without concrete callable contract content.

Page payload:
{{page_content}}
''')
