# 多文档测试用例生成优化需求文档

## 1. 文档定位

本文档是对以下两份已有文档的补充和细化：

- `agent/gen_auto_case_multi_doc/多文档测试用例生成执行需求文档.md`
- `agent/gen_auto_case_multi_doc/多文档测试用例生成重构方案.md`

本文档不替代原有文档，主要用于沉淀当前代码改造后暴露出的缺陷、风险点、后续优化目标、优先级和验收标准，作为下一阶段持续重构的直接输入。

## 2. 当前范围定义

当前 `gen_auto_case_multi_doc` 已经具备以下能力：

1. 保留单文档执行能力。
2. 支持 `kbSpaceFileId` 传入多个文件 ID。
3. 支持 `web` / `api` / `auto` 三种模式。
4. 支持生成三张核心表：
   - `kb_gen_test_module`
   - `kb_gen_test_module_design`
   - `kb_gen_test_case`
5. 支持 Step2 / Step3 继续复用当前主链路。
6. 支持来源格式从 `PageN` 扩展为 `fileName#PageN`。

当前下一阶段优化的重点，不是重写全流程，而是提升多文档融合质量、可维护性、可追溯性和稳定性。

## 3. 当前主要问题总览

### 3.1 Step1 合并仍然偏“字段级拼接”，不是“语义级整合”

当前多文档融合的核心逻辑仍主要依赖以下字段：

- `section`
- `title`
- `source_type`
- `requirement_type`

问题：

1. 同义需求若标题不同，可能无法合并。
2. 标题相近但语义不同，存在误合并风险。
3. 目前更接近“页摘要合并”，还不是“系统级功能意图合并”。
4. 补充文档对主需求的增强方式较粗，容易把补充信息堆到描述里，而不是精确挂到某一个功能点。

### 3.2 Step1 的补充文档增强不够精准

当前 `Background` / `Compliance` / `Other` 类内容，更多是摘要后拼接到功能描述中。

问题：

1. 无法保证某条业务规则准确归属到对应功能。
2. 无法区分“应增强已有功能”和“应新增独立功能”。
3. 容易造成描述膨胀，但对后续 Step2 / Step3 提升有限。

### 3.3 思维导图当前主要是展示结果，不是中间结构核心

当前 Mermaid 思维导图来源于 `combined_result` Markdown。

问题：

1. 它不是多文档融合的唯一事实来源。
2. 它不直接驱动 Step1 合并结果。
3. 它与最终真正送入 Step1 的融合输入并非完全一一对应。

### 3.4 `testability` 目前没有真正进入主过滤链路

页摘要中已经定义了 `testability` 字段，但主流程并未将其作为强过滤条件。

问题：

1. 目录页、弱相关说明页、管理信息页仍可能进入后续流程。
2. Step1 输入噪声偏多。
3. 会放大 Step1 token 消耗和误识别概率。

### 3.5 多文档页码语义仍不够清晰

当前多文档模式下，页码和来源使用策略为：

- `page` 继续表示文档内页码
- `source` / `source_ref` 可能为 `fileName#PageN`

问题：

1. `page` 本身不具备跨文档唯一性。
2. `pages` 入参仍是全局页筛选，不是“按文档维度筛页”。
3. 某些下游逻辑仍存在先取 `page` 再推断来源的倾向。

### 3.6 `platform` 当前仍是写死值

当前 `platform` 固定为 `DEMO-platform`。

问题：

1. 这只是阶段性兼容方案。
2. 后续必须切换为外部传值或上游写入。
3. 当前写死实现需要统一清理，避免后续多点修改。

### 3.7 多来源追溯能力仍不完整

当前虽然已经支持 `fileName#PageN` 回查，但表结构层面仍偏兼容旧逻辑。

问题：

1. `documentId` 仍主要承载兼容意义，不足以表达真实多来源。
2. `main_module_source` / `submodule_source` / `source_value` 同时承担展示和追溯两种职责，语义较混。
3. `evidence_summary` 尚未真正落地使用。

### 3.8 脚本实现中正则和重复校验偏多，增加语法风险

当前入口和若干处理中存在较多 `replaceAll`、正则拆分、重复空值校验。

问题：

1. AgentFlow/Groovy 嵌套字符串环境下，正则转义容易出错。
2. 多层 `replaceAll` 可读性差，调试成本高。
3. 同一参数已在上游校验过，子块再次校验会增加冗余。

### 3.9 Step1 为避免超 token 进行输入压缩，存在质量折损风险

当前 `agt_test_doc_func_modules.txt` 和 `agt_test_doc_func_modules_api.txt` 已经通过“先压缩输入、再调用 LLM”的方式降低 Step1 超 token 风险。

问题：

1. 长 `description` 被截断后，模块识别可能遗漏后半段关键约束。
2. `business_rules` / `acceptance_criteria` 仅保留前若干条时，靠后的高价值规则可能丢失。
3. 当同一功能的关键差异藏在长文本后半段时，Step1 可能合并过度。
4. 当前这种压缩方式更偏“止血方案”，不是长期稳定方案。
5. 如果入口批量仍然过大，即使做了压缩，Step1 仍可能出现 token 紧张或输出发散。

## 4. 下一阶段优化目标

### 4.1 总体目标

在不破坏现有单文档能力、不重写 Step2 / Step3 主逻辑的前提下，完成以下提升：

1. 让 Step1 真正基于“系统级功能意图”进行多文档融合。
2. 让补充文档准确增强已有功能，而不是简单堆描述。
3. 让来源追溯字段职责清晰、稳定。
4. 让脚本实现方式更简单、可维护、低语法风险。

### 4.2 核心原则

1. 单文档逻辑必须持续可用。
2. 多文档优化优先落在 Step1 前和 Step1 内。
3. Step2 / Step3 只做必要兼容，不做大规模重写。
4. 能不用正则就不用正则。
5. 能复用已有字段就不新增字段。
6. 参数数量不增加，继续复用 `kbSpaceFileId`。

## 5. 结构化优化需求

### 5.1 优化需求 A：重构 Step1 融合模型

#### 目标

将当前“页摘要列表 -> 直接分类 -> 合并”的模式，升级为“页摘要原子项 -> 功能候选项 -> 系统级融合项”。

#### 需求内容

1. 在 Step1 内部引入“候选功能项”概念。
2. 每个候选功能项至少包含：
   - `platform`
   - `source_type`
   - `section`
   - `title`
   - `description`
   - `business_rules`
   - `acceptance_criteria`
   - `dependencies`
   - `page`
   - `source_ref`
   - `fileName`
3. 多文档融合时，不直接按页合并，而是先形成候选功能项集合。
4. 合并依据从“标题相同”升级为“功能意图相同 + 规则一致或互补”。
5. 对补充文档的默认动作改为：
   - 优先增强已有候选项
   - 仅在存在独立可测试能力时新增候选项

#### 验收标准

1. A 文档单独可生成 10 条 Step1 数据时，加入 B/C 补充文档后，结果应优先表现为原 10 条增强，只有确实新增功能时才增加条数。
2. 不允许出现“按页数线性膨胀 Step1 数量”的结果。

### 5.2 优化需求 B：补充文档精确挂载

#### 目标

让背景、合规、其他类内容不再粗暴拼接到 `description`，而是优先精确增强已有功能项。

#### 需求内容

1. `Background` 类内容优先补充：
   - 业务上下文
   - 角色/权限
   - 上下游依赖
2. `Compliance` 类内容优先补充：
   - 规则约束
   - 校验要求
   - 审计/权限/合规验证点
3. `Other` 类内容默认不直接生成功能项。
4. 无法挂载到任何功能项的补充内容，进入未映射集合，不直接进入 Step1 产出。

#### 验收标准

1. Step1 的 `description` 不应无限膨胀。
2. 业务规则优先进入 `business_rules` 或 `acceptance_criteria`。
3. 无法映射的补充信息应可定位，不得静默丢失。

### 5.3 优化需求 C：让思维导图与融合结果对齐

#### 目标

思维导图不再只是 `combined_result` 的附属展示，而要与最终融合后的功能结构一致。

#### 需求内容

1. 思维导图应基于 Step1 最终融合结果生成。
2. 图中节点应反映：
   - 系统
   - 主模块
   - 子模块
   - 关键规则/来源
3. 图不作为唯一数据源，但必须与最终 Step1 结果一致。

#### 验收标准

1. 图中模块数量和 Step1 产物结构基本对应。
2. 不允许出现图和最终模块结构明显不一致的情况。

### 5.4 优化需求 D：将 `testability` 接入主链路

#### 目标

将不可测试或弱相关页尽早过滤。

#### 需求内容

1. 页摘要后增加 `testability` 过滤。
2. 以下内容默认不进入 Step1：
   - 目录页
   - Jira/任务管理元数据页
   - 无测试语义的纯背景页
   - 与系统功能无关的说明页
3. 对“有少量可测试信息”的页面，允许保留结构化摘要，但不能整页进入功能生成输入。

#### 验收标准

1. Step1 输入项数量明显下降但功能覆盖不下降。
2. 非功能性噪声页不应生成模块。

### 5.5 优化需求 E：明确页码与来源字段职责

#### 目标

彻底区分“展示页码”和“追溯来源”。

#### 需求内容

1. `page` 仅表示文档内页码。
2. `source` / `source_ref` 统一承担跨文档追溯职责。
3. 多文档模式下，Step1 输出优先使用 `fileName#PageN`。
4. Step2 / Step3 只把 `page` 当辅助信息，不再把它视为跨文档唯一定位键。

#### 验收标准

1. 任一设计点或用例都能回查到原始文档页。
2. 多文档下不再因同名页码冲突而取错内容。

### 5.6 优化需求 F：统一 `platform` 注入方式

#### 目标

消除写死平台值的分散实现。

#### 需求内容

1. 当前阶段继续允许固定值运行。
2. 代码层必须收敛为单点注入。
3. 后续切换外部传值时，不允许多处搜索替换。

#### 验收标准

1. `platform` 的赋值位置明确、唯一。
2. Step1 / Step2 / Step3 都只透传，不自行生成。

### 5.7 优化需求 G：增强多来源追溯字段设计

#### 目标

让表数据既兼容旧逻辑，又能表达多文档真实来源。

#### 需求内容

1. 继续复用现有字段：
   - `main_module_source`
   - `submodule_source`
   - `source_value`
2. 评估是否正式启用：
   - `evidence_summary`
3. `documentId` 保持兼容字段定位，不再承载完整多来源语义。

#### 验收标准

1. 表结构不盲目膨胀。
2. 追溯链路能覆盖模块、设计点、用例三层。

### 5.8 优化需求 H：脚本实现去正则化、去冗余化

#### 目标

降低 AgentFlow/Groovy 字符串脚本的语法风险。

#### 需求内容

1. `kbSpaceFileId` 解析仅支持英文逗号分隔，不做数组、JSON、方括号兼容。
2. 已在上游校验过的空值，下游不重复校验。
3. 能用 `contains`、`split`、`trim` 的场景，不使用正则。
4. 仅在必须做语义识别时保留正则。

#### 验收标准

1. 入口解析逻辑明显简化。
2. 高风险 `replaceAll` 使用点显著减少。
3. 语法错误率下降。

### 5.9 优化需求 I：Step1 token 控制与质量平衡

#### 目标

在控制 Step1 token 消耗和输出稳定性的同时，尽量减少对模块识别质量和规则完整性的负面影响。

#### 需求内容

1. 明确区分“临时压缩止血方案”和“正式结构化瘦身方案”。
2. Step1 输入不要长期依赖简单截断 `description` 保命。
3. 应优先在页摘要阶段完成结构化瘦身，再将精简后的高价值字段送入 `func_modules`。
4. 对以下高价值信息建立优先保留策略：
   - 权限/访问控制
   - 状态流转
   - 多语言要求
   - 异常处理
   - 接口契约约束
5. `business_rules` 和 `acceptance_criteria` 不能只按顺序简单截前 N 条，后续应支持按重要度保留。
6. Step1 的 chunk 大小应结合实际 token 风险重新评估，不能固定依赖当前较粗的批量策略。
7. 需要区分：
   - 模块识别所需信息
   - 规则增强所需信息
   避免所有信息一次性塞入同一个 LLM 调用。

#### 验收标准

1. Step1 在多文档场景下不因输入过大频繁超 token 或输出发散。
2. 压缩后主模块/子模块骨架质量保持稳定。
3. 关键规则不会因简单截断被系统性丢失。
4. Step1 的输入控制方式从“字符串截断”逐步过渡为“结构化信息筛选”。

## 6. 优先级建议

### P0：必须优先处理

1. Step1 融合模型重构
2. 补充文档精确挂载
3. `testability` 接入主过滤链路
4. `page` / `source_ref` 职责统一
5. 入口和关键链路去正则化

### P1：第二阶段处理

1. 思维导图与最终融合结果对齐
2. `platform` 单点注入收敛
3. `evidence_summary` 正式启用
4. Step1 token 控制从临时压缩方案升级为结构化瘦身方案

### P2：后续增强

1. 更细粒度的来源字段设计
2. 更强的语义聚类策略
3. 更精细的补充信息冲突检测

## 8. 实施顺序建议

### 第一阶段

1. 简化入口参数解析逻辑
2. 清理重复校验
3. 接入 `testability` 过滤
4. 收敛 `source_ref` 输出格式
5. 先保证 Step1 在大输入下不超 token

### 第二阶段

1. 重构 Step1 融合模型
2. 增加补充文档精确挂载逻辑
3. 调整 Step1 输出与思维导图一致性
4. 将 Step1 输入控制从简单截断升级为结构化瘦身

### 第三阶段

1. 收敛 `platform` 注入
2. 启用 `evidence_summary`
3. 补齐 Step2 / Step3 在多来源追溯上的细节

## 9. 结果要求

完成上述优化后，目标状态应为：

1. 单文档继续按原逻辑稳定执行。
2. 多文档不按文档数或页数膨胀 Step1 数量。
3. 补充文档主要用于增强已有功能，而不是制造大量新模块。
4. 三张核心表仍保持原有产出形式。
5. 后续代码修改基于本文件与原两份文档联合执行。
