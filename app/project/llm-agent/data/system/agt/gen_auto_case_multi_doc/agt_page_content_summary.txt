define('tpl_page_summary', '''
You are extracting structured requirement items from one page slice of multi-document requirement material.

Return JSON array only.

Hard rules:
1. Return one JSON array only. No markdown, no explanation, no comments.
2. `requirement_type` must use one fixed English enum only: `Background`, `Functional`, `Compliance`, `Other`.
3. All user-visible text fields other than fixed enums must be written in {{lang_output_text}}.
4. If this page has no clear, testable, traceable requirement content, return [].
5. Do not fabricate title, section, business rule, acceptance criteria, or dependency values.
6. Ignore TOC, navigation, issue metadata, sprint/task management text, and low-value project admin content.
7. Merge duplicate statements within the same page instead of repeating them.

Output schema:
```json
[{
  "page": 1,
  "requirement_type": "Functional",
  "section": "Module or business area",
  "title": "Requirement title",
  "description": "One concise requirement description",
  "business_rules": ["Rule 1"],
  "acceptance_criteria": ["Criterion 1"],
  "dependencies": ["REQ-001"],
  "testability": true
}]
```

Field guidance:
- `section`: stable business area or feature area, not a document formatting label.
- `title`: the smallest clear testable requirement title from this page.
- `description`: concise summary of behavior, constraint, or user-visible flow.
- `business_rules`: only hard rules, validations, permissions, state transitions, or constraint statements.
- `acceptance_criteria`: directly testable expected outcomes.
- `dependencies`: only explicit dependencies or referenced requirements.
- `testability`: false only when the page is mostly background, glossary, TOC, management metadata, or otherwise not directly testable.

Page payload:
{{page_content}}
''')
