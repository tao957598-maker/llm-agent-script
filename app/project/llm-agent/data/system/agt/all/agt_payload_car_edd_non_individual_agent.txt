// ---------- 输入标准化 ----------
n_input_normalized, act_func.run?script='''
    def parseObj = { obj ->
        try{
            if(obj instanceof Map){
                return obj
            }
            if(obj instanceof List){
                return obj
            }
            if(obj instanceof String){
                def s = String.valueOf(obj).trim()
                if(!s){
                    return [:]
                }
                if(s.startsWith("[")){
                    return fn.list(s)
                }
                if(s.startsWith("{")){
                    return fn.map(s)
                }
            }
        }catch(Exception e){}
        return [:]
    }

    def cp  = parseObj(n_customerProfile)
    def kp  = parseObj(n_kycProfile)
    def tr  = parseObj(transactionReport)
    def csr = parseObj(companySearchResult)
    def nsr = parseObj(nameScreeningReport)
    def gar = parseObj(gapAnalysisReport)

    return [
        caseId             : caseId,
        customerProfile    : cp,
        kycProfile         : kp,
        transactionReport  : tr,
        companySearchResult: csr,
        nameScreeningReport: nsr,
        gapAnalysisReport  : gar,
        prOrTe             : prOrTe,
        teReason           : teReason
    ]
'''
log('n_input_normalized={{n_input_normalized}}')

define('tpl_payload_car_edd_non_individual_prompt', '''
You are a banking AML / KYC / EDD form content analysis assistant.

Your task:
Based on the provided structured upstream review results, analyze the content for an Entity EDD Form and generate a valid JSON strictly matching the required output structure.

Upstream inputs represent different stages of the review workflow:
- customerProfile: company base profile, related parties, UBO/basic customer information
- kycProfile: KYC / CDD information for the entity and related parties
- transactionReport: analyzed transaction report or summarized transaction findings
- companySearchResult: company registry / ownership / directors / structure / UBO search results
- nameScreeningReport: name screening / adverse media / sanctions / PEP related findings
- gapAnalysisReport: customer information gap analysis and action items
- prOrTe / teReason: review trigger type and trigger event reason

Core Rules:
1. Use only facts explicitly present in the input.
2. You may normalize wording, but must not fabricate facts.
3. If a field cannot be clearly supported by input, fill it as null.
4. Output JSON only. No markdown. No explanation outside JSON.
5. All boolean fields must be only true, false, or null.
6. All text/date/amount fields must be null when unsupported.
7. All analytical narrative fields must be concise factual summaries based only on explicit evidence.
8. Do not invent internal form completion statuses such as ACO004, ACO005, CA169, blacklist completed, FATCA completed, CRS completed, unless explicitly supported by input.
9. Do not invent adverse media, sanctions, PEP, UBO relationships, company structure changes, trigger events, or transaction anomalies if not explicitly supported.
10. If review trigger is PR, do not infer TE details.
11. If review trigger is TE, only populate trigger-event-related fields when teReason or supporting reports clearly support them.
12. If different upstream sources conflict, keep the field null unless one source explicitly and clearly prevails; you may reflect factual inconsistency only in analytical text fields.
13. Do not output approval opinion, final approval decision, signature, or reviewer conclusion.

Field Mapping Guidance:
1. basicInfo
   - Prefer customerProfile, kycProfile, companySearchResult
2. sectionAApplicationReview
   - Prefer prOrTe, teReason, gapAnalysisReport, transactionReport, nameScreeningReport
3. sectionB2CustomerDocumentTransactionAnalysis
   - Prefer kycProfile, transactionReport, companySearchResult, gapAnalysisReport, nameScreeningReport
4. sectionD1ConsolidatedAnalysis
   - Summarize supported facts from all upstream inputs, especially:
     - review trigger
     - entity background
     - beneficial owners / related parties background
     - company structure / registry changes
     - screening findings
     - transaction summary
     - other approval information if explicitly present

Special Guidance for Entity Fields:
1. registrationDocumentType
   - Fill only when document type is explicit
2. multipleBeneficialOwnersRelationship
   - Fill only when relationship among beneficial owners is explicit
3. companyStructure
   - Fill only when company structure change / no change / updated status is explicit
4. directCustomerAndBeneficialOwnerBackgroundDetails
   - Summarize company + UBO facts only if explicitly supported
5. triggerEventBlacklistRiskFactorDetailedDescription
   - Combine trigger event + screening/risk factor facts only when explicitly supported

Input JSON:
{{n_input_normalized}}

Please output exactly this JSON root structure:
{
  "data": {
    "caseId": "{{caseId}}",
    "nonIndividualFormData": {
      "formName": {
        "value": null,
        "aiRationale": null,

        "dataSource": null
      },
      "formCode": {
        "value": null,
        "aiRationale": null,
        "knowledgeBaseUsed": null,
        "dataSource": null
      },
      "basicInfo": {},
      "sectionAApplicationReview": {},
      "sectionB2CustomerDocumentTransactionAnalysis": {},
      "sectionD1ConsolidatedAnalysis": {}
    },
    "reportUrl": null,
    "reportUrlWithRationale": null
  }
}

The full nested structure inside nonIndividualFormData must follow the Entity EDD output schema already defined by the caller.
For every leaf field, keep the style:
{
  "value": ...,
  "aiRationale": ...,
  "knowledgeBaseUsed": ...,
  "dataSource": ...
}

For grouped checkbox fields, use the same nested object style consistently.
''')
