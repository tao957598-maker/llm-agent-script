# 多文档测试用例生成执行需求文档

## 0. 本次审核结论

当前执行策略需要收紧并统一：目标不是做一个独立的新项目流程，而是在保留当前单文档能力的基础上，让一个系统下的多个文档合并生成当前三张核心表：

```text
kb_gen_test_module
kb_gen_test_module_design
kb_gen_test_case
```

对应导出仍是：

```text
Test Module
Test Module Design
Test Case
```

第一版不重写现有主链路，尤其不重写 Step2 设计点生成和 Step3 用例生成。多文档能力主要加在 Step1 之前：读取多个文档、逐页摘要、按系统和功能意图归并，再生成一套 Step1 输入。

参数也要收紧。第一版不新增 `folderId/projectId/kbSpaceFileIds/primaryDocumentId`，直接复用当前 `kbSpaceFileId` 参数，让它兼容单个文件 ID 和多个文件 ID：

```text
单文档：kbSpaceFileId = file_001
多文档：kbSpaceFileId = file_001,file_002,file_003
```

如果未来 UI 支持选择文件夹，也建议在进入本 Agent 前先把文件夹展开为文件 ID 列表，再传给 `kbSpaceFileId`。

系统名称由本 Agent 前置流程提供，并统一写入 `platform`。本 Agent 只读取和传递 `platform`，不再让 AI 生成或猜测系统名。

强制规则：

```text
platform = 前置流程提供的系统名称
Step1 输出中的 platform 必须等于该值
agt_test_doc_func_modules / agt_test_doc_func_modules_api 不允许根据文档标题自行生成 platform
如果模型返回的 platform 为空或不一致，父流程必须用前置 platform 覆盖
```

第一版核心目标是：

```text
把多文档内容在 Step1 之前融合好，
再转换成当前 Step1 可以消费的数据结构，
让后续 Step1 / Step2 / Step3 尽量沿用现有逻辑。
```

最终完整流程定义为：

```text
kbSpaceFileId 一个或多个文件 ID
  -> 解析 file_id_list
  -> 读取 parse 表中这些文件的页面
  -> 按页摘要和分类
  -> 识别文档角色和核心文档集
  -> 按 platform + 功能意图归并
  -> 生成一套 functional_list / api_list
  -> 复用 Step1 生成 kb_gen_test_module
  -> 复用 Step2 生成 kb_gen_test_module_design
  -> 复用 Step3 生成 kb_gen_test_case
  -> 导出三份 xlsx
```

因此第一版修改边界是：

```text
入口参数
文档清单
多文档页读取
多文档页摘要
多文档分类和归并
combined_result
Step1 输入准备
```

第一版不主动修改：

```text
Step2 设计点生成提示词
Step3 用例生成提示词
case 去重和写表主逻辑
当前单文档 resume 主逻辑
```

但是有一个必须处理的兼容点：当前 Step2 / Step3 会继续用前一步数据中的页码回查原文。如果多文档模式输出裸 `PageN`，后续会不知道这是哪份文档的页。因此第一版要在两种策略中选一种：

| 策略 | 修改范围 | 优点 | 缺点 |
| --- | --- | --- | --- |
| A. 主文档兼容策略 | 内部选择 `primary_document_id = file_id_list[0]` 给 Step2/Step3 回查，其他文档内容在 Step1 前吸收到摘要和模块描述中 | 最小改动，最安全 | Step2/Step3 不能重新拉取辅助文档原文 |
| B. 来源解析适配策略 | 给 Step2/Step3 增加一个轻量 `source_refs -> pages` 解析节点，生成逻辑不变 | 能真正跨文档回查 | 需要小改 Step2/Step3 的取页部分 |

推荐执行顺序：先做策略 A，跑通多文档融合；再做策略 B，增强跨文档追溯。

### 0.1.3 parse 表字段要求

该表不只服务 case 生成，因此不要求新增 case 专用字段。只要求能获取以下信息：

| 字段 | 是否必需 | 定义 |
| --- | --- | --- |
| platform 或 system_name | 后续新增 | 系统名称。进入 Agent 后统一映射为 `platform` |
| document_id / documentId | 已有 | 文件 ID，对应当前 `documentId` |
| fileName | 已有 | 文件名，用于来源追溯和文档角色识别 |
| page | 已有 | 文档内页码，不做跨文档唯一标识 |
| content / analysis_content | 已有 | 页面正文或解析正文 |

## 1. 执行原则

本执行需求文档用于后续逐步修改 `agent/gen_auto_case` 下的 AgentFlow 流程。

执行时遵循：

1. 每一步只改一个明确能力点。
2. 每一步必须保持单文档兼容。
3. 每一步都要能通过 AgentFlow 语法检查。
4. 涉及数据结构变化时，先在内存变量中兼容，再逐步落表。
5. 不一次性重写主流程，优先在现有节点旁边扩展。
6. 多文档能力以 `platform`、`document_id` 和 `source_ref/global_page_ref` 为核心。
7. Mermaid 思维导图从结构化数据生成，不作为唯一生成依据。

### 4.8 新增 Step1 输入归并

第一阶段最重要的是把多文档内容归并成 Step1 能消费的输入。

以三个文档为例，如果一共从 parse 表读取到 30 页数据，处理方式不是直接把 30 页原文一次性塞给 Step1，也不是每个文档分别生成一套 Step1 后再拼接，而是采用“自动识别核心文档集 + 补充文档增强”的方式：

```text
30 页 parse 数据
  -> 逐页摘要，得到 page_summary_list
  -> 按 requirement_type/source_type/document_type 分类
  -> 先自动识别核心文档集
  -> 从核心文档集生成 Step1 候选功能基线
  -> 再用补充文档增强、约束、少量新增
  -> 生成一套 multi_doc_step1_input
  -> 转成现有 functional_list / api_list
  -> 调用现有 Step1 模块生成
```

合并发生在“页摘要之后、Step1 之前”。

核心原则：

```text
Step1 数量由自动识别出的核心文档集决定上限基线。
补充文档默认不扩张模块数量，只补充已有候选功能。
只有补充文档中出现主需求文档完全没有覆盖、且明确可测试的独立功能时，才允许新增 Step1 候选项。
```

例如：

```text
B = 补充文档
C = 补充文档
A = 被系统自动识别为核心文档之一，单独可生成 10 条 Step1 候选数据

合理结果：
10 条主候选 + 0~2 条确实新增的独立功能 = 10~12 条

不合理结果：
A/B/C 各自生成后简单相加，膨胀到 30 条
```

原因：

1. 直接把 30 页原文交给 Step1，token 大、噪声多、背景文档会干扰模块拆分。
2. 每个文档各跑 Step1 再合并，容易产生重复模块。
3. 页摘要先把每页压缩成结构化数据，后续才能按业务意图归并。

新增变量：

```text
multi_doc_step1_input
multi_doc_functional_list
multi_doc_api_list
multi_doc_background_context
multi_doc_business_rules
```

归并规则：

1. 功能需求和 API 文档内容进入 `multi_doc_functional_list` 或 `multi_doc_api_list`。
2. 业务规则不单独直接生成模块，而是附加到相关功能摘要的 `business_rules/acceptance_criteria/description` 中。
3. 背景文档默认不生成模块，只作为 `background_context` 补充术语、角色、业务场景。
4. 合规文档可作为约束进入相关功能摘要。
5. 第一版不要求生成 requirement atoms。
6. 合并的目标不是“把每个文档都生成一套 Step1”，而是“把同一系统下的多文档合成一套 Step1 输入”。
7. 同一功能点在多个文档出现时，只保留一个 Step1 候选项，但合并所有来源文档、页码、业务规则和验收标准。
8. 如果某页只有背景、目录、术语或说明，不直接进入 Step1 候选项，只作为上下文补充。
9. API/接口页如果能对应某个功能点，则挂到该功能点；如果是独立接口能力，则进入 `multi_doc_api_list`。
10. 输出给 Step1 的每条数据必须带 `platform`，且值使用前置流程提供的系统名。
11. 补充文档新增 Step1 候选项必须满足：主需求文档无对应功能、内容可测试、不是纯背景说明、不是同义重复、不是只补充字段/规则。
12. 对补充文档中的接口、角色、报表、权限、流程说明，优先并入已有候选项的 `business_rules/acceptance_criteria/description/source`，不要直接拆成新模块。
13. 文档顺序不作为主锚点依据，必须先做文档角色识别和内容评分。
14. 允许多个核心文档共同构成主锚点集，不强制全系统只有一个主文档。

### 4.8.1 数据量控制策略

为了避免输入和输出数据量过大，第一版不把所有页摘要原样传给 Step1。需要在 Step1 前做压缩和裁剪：

1. 主需求文档摘要保留完整候选信息。
2. 补充文档摘要只保留与候选功能相关的规则、接口、角色、报表、异常、验收点。
3. 每个候选功能最多携带有限数量的补充证据，例如每类补充保留 3~5 条关键点。
4. 背景文档只保留系统级术语、角色、流程，不进入每个候选功能，除非明确关联。
5. 无法关联到任何候选功能的补充内容进入 `unmapped_supplement_list`，不直接生成 Step1。

### 4.8.2 多核心文档处理方式

当系统里有多个都很像主文档的文件时，不要强行选一个唯一主文档。更合理的是：

1. 先给每个文档打 `doc_score`。
2. 取分数高的一组进入 `core_doc_set`。
3. `core_doc_set` 可以有多个文件。
4. 每个业务域或功能域再从 `core_doc_set` 里选该域的主证据文档。
5. 补充文档只增强对应域，不抢主位。
6. 如果两个文档高分但覆盖不同业务域，它们都可以同时是核心文档。

推荐中间结构：

```json
{
  "platform": "CSS-CL",
  "base_requirement": {
    "title": "GM FX Prompt maintenance",
    "description": "主需求文档中的功能描述",
    "source": "A.docx",
    "page": "3"
  },
  "supplements": {
    "business_rules": ["最多保留关键规则"],
    "api_contracts": ["最多保留关键接口约束"],
    "roles_permissions": ["最多保留关键角色权限"],
    "reports": ["最多保留关键报表检查点"],
    "background": ["必要术语或流程上下文"]
  },
  "source_refs": ["A#Page3", "B#Page8", "C#Page2"],
  "merge_action": "enhance_existing"
}
```

`merge_action` 可选值：

```text
enhance_existing    补充已有主需求候选项
create_new          补充文档中发现明确独立功能，新增候选项
ignore_context      只作为背景上下文，不进入 Step1
needs_review        无法判断是否应新增，进入待确认
```

给 Step1 的输入仍应尽量保持当前 `functional_list` 的结构，这样可以复用：

```text
agt_test_doc_func_modules
agt_test_doc_func_modules_api
```

| --- | --- | --- |
| 多文档 token 过大 | LLM 生成慢或失败 | 按需求原子和模块上下文包分批 |
| 文档类型识别错误 | 规则权重不准 | 第一版规则识别，后续 LLM 分类 |
| PageN 冲突 | 来源错误 | 强制使用 `DOC#PageN` |
| Mermaid 生成失败 | 展示失败 | fallback 到旧导图或静态导图 |
| 新字段破坏旧表 | 写入失败 | 第一阶段先内存兼容，落表分阶段 |
| 子模块重名 | 续跑错误 | 引入 `submodule_id` |
| 背景文档生成无效 case | case 噪声 | 背景默认 `testability=false` |

## 17. 推荐执行顺序

严格按以下顺序执行：

1. 复用 `kbSpaceFileId`，新增内部解析逻辑并生成 `input_mode/project_id/file_id_list`。
2. 构建 `document_manifest`。
3. 改造页读取，生成带文档身份的 `n_kb_data`。
4. 改造 page summary 输出补字段。
5. 改造 `combined_result` 文档分组。
6. 新增多文档归并逻辑，生成 `multi_doc_step1_input`。
7. 将 `multi_doc_step1_input` 转换成现有 Step1 可消费的 `functional_list/api_list` 结构。
8. 复用现有 Step1 模块生成，先不改 Step2 / Step3。
9. 跑通单文档回归和多文档第一版。
10. 再新增 requirement atoms。
11. 再新增 atom merge。
12. 再新增 requirement graph。
13. 再从 graph 生成 Mermaid。
14. 再构建 module context packages。
15. 再新增按 source_refs 跨文档查页。
16. 再轻量改造 Step2 取页逻辑，生成主逻辑不变。
17. 再轻量改造 Step3 取页逻辑，生成主逻辑不变。
18. 改造 resume。
19. 改造导出和统计。
20. 补全测试和回归。

## 18. 第一轮最小可交付范围

为了降低风险，第一轮建议只做：

1. 新增多文档输入参数。
2. 构建 `document_manifest`。
3. 页读取支持多个文档。
4. 页摘要补充文档身份。
5. `combined_result` 按文档分组。
6. 新增多文档归并为 Step1 输入的逻辑。
7. Mermaid 仍可使用旧 `tpl_txt_to_mermaid`。
8. Step1 / Step2 / Step3 暂时保持旧逻辑，只验证多文档读取、归并和单文档兼容。

第一轮不解决完整跨文档原文回查；辅助文档信息先通过 Step1 前的归并进入模块描述和摘要上下文。完整 `source_refs` 跨文档回查放到第二轮。

第一轮完成后，再进入需求原子和图谱。
