### 🛠️ 优化后的提示词：功能分解（Functional Decomposition）/模块化设计（Modular Design）根据类型是 Functional

**任务目标**：
对系统或产品进行**一级功能分解**，侧重于**人机交互单元**的识别与拆解，以“功能页面”或“具体操作界面”为核心视角，构建清晰、精细的功能模块结构。

---

### ✅ 拆解原则
按EARS方案(普通需求/事件需求/状态需求/可选需求/异常需求/复杂需求/模糊需求)做功能拆解

1. **以产品界面或业务操作为主线视角**，而非抽象架构。
2. **每个模块代表一个清晰、独立的用户交互单元**，通常对应于一个操作页面或一个功能触发点。
3. **模块命名需细化到页面级别或流程节点级别**
4. **使用连字符“-”连接上下文与具体页面**，体现模块所在路径或导航关系，保留模块层级语义。
5. **功能颗粒度需细到“界面/操作流程最小单元”**：

   * 弹窗、侧边栏、表单提交等重要交互单元也可单独视作模块
6. **明确区分界面模块与非界面模块**：

   * 非界面模块请显式标注，如：`【非界面模块】任务调度逻辑`
7. **严格遵循命名模板**：

   ```
   [一级功能模块名称] - [页面或操作单元名称]
   ```
8. 标记来源哪些页码例如: Pagexx,Pagexxx；同时输出 `page` 字段，保留原始页码数字，不要把页码只写进 `source`。
语言风格: 遵从文档原始术语（如标题中是英文的）保持一致更好
9. 若原文出现“权限/注册检查/feature registration/access control”等规则，必须在模块描述中保留该校验语义，不可丢失。
11. 不得生成与原文无证据的模块；每个子模块必须可追溯到输入文本中的明确语句。
12. 若仅为同一页面的同义称呼/渠道前缀差异（如不同渠道或终端对同一操作步骤且规则一致），优先合并为一个子模块，并在 description 中注明适用渠道；仅当流程、输入或规则确实不同才拆分多个子模块。
13. 控制粒度：优先覆盖关键流程与规则，避免“仅改写措辞”的平行子模块重复输出。
---

### 🔁 辅助输入（建议提供以增强一致性）：

Field semantics refinement:
- For a single-function requirement document, prefer the business function name / Chinese feature name / feature title as the top-level module name. Do not default the top-level module name to a generic page name unless the source text clearly defines a page module.
- `platform` should represent the system, channel, product container, or runtime platform. Do not mix the feature/function name into `platform`.
- If both a feature name and a system/channel name appear in the document, keep them semantically separated: feature name for module naming, system/channel name for `platform`.
Additional Web decomposition guardrail:
- Do not split modules only by wording variants, display labels, input value categories, or channel/context labels when they describe the same executable test intent.
- Keep one submodule and describe supported variants/contexts in `description` only when the page state, primary user action, validation rules, actor responsibility, and business outcome are still the same.
- Do split when the document shows a different primary action or entry point on the same page, such as upload, download, submit, confirm, navigate to next step, open details, trigger validation, or any other independently testable control.
- Do split when the document shows a different UI/page state, different required fields, different validation rules, different actor/system responsibility, different business stage, or materially different downstream outcome.
- Do not merge page entry/access check logic with subsequent in-page action controls when they are separately testable.
- Do not merge static display/content areas with actionable controls when the controls trigger their own flow or system behavior.
