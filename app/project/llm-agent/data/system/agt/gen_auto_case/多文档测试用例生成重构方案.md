# 多文档测试用例生成重构方案

## 0. 本次审核后的落地边界

本方案需要优先保证当前单文档主流程不被破坏。也就是说，现有 `kbSpaceFileId -> 页摘要 -> combined_result -> Step1 模块 -> Step2 设计点 -> Step3 用例` 的链路必须继续存在，单文档场景默认仍走当前逻辑。

参数设计也需要保持简单。第一版不新增 `projectId/folderId/kbSpaceFileIds/primaryDocumentId` 等入口参数，直接复用当前 `kbSpaceFileId`：

```text
单文档：kbSpaceFileId = 一个文件 ID
多文档：kbSpaceFileId = 多个文件 ID，逗号分隔或数组形式
```

如果后续支持选择文件夹，也建议在进入本 Agent 前先把文件夹展开成多个文件 ID，再传给同一个 `kbSpaceFileId` 参数。这样当前 UI 和调用链路都不用大改。

多文档的聚合维度不再额外定义为 project，而是直接按“系统”聚合。系统名称优先复用现有 `platform` 字段：

```text
platform = 系统名称
```

`platform` 由本 Agent 的前置流程提供，本 Agent 不负责识别系统名，也不再让 AI 根据文档标题生成 platform。后续如果 parse 表新增系统字段，也建议与 `platform` 保持语义一致，例如 `system_name` 或 `platform`。执行链路中可以内部统一成 `platform`，最终写入 `kb_gen_test_module`、`kb_gen_test_module_design`、`kb_gen_test_case`。

强制规则：

```text
platform 必须取前置流程提供的值
Step1 模型返回 platform 后，父流程仍要用前置 platform 覆盖
Step2 / Step3 继续从前一步数据继承 platform
```

多文档第一版不建议直接重写 Step2 和 Step3。更稳妥的边界是：

```text
多文档输入
  -> 多文档读取
  -> 多文档页摘要
  -> 多文档内容归并
  -> 生成 Step1 仍可消费的输入数据
  -> 复用现有 Step1 / Step2 / Step3 主逻辑
```

因此，第一阶段主要修改范围应截止到“Step1 模块数据生成之前”。Step1 之后的模块生成、设计点生成、用例生成，可以先保持当前业务逻辑不变。

但需要注意一个关键点：当前 Step2 / Step3 会根据前一步产物中的 `page/main_module_source/submodule_source` 再回查 `kbs_doc_page_parse`。如果多文档后 Step1 输出的来源仍是裸 `Page1`，就会出现多个文档页码冲突。因此第一版虽然不重写 Step2 / Step3 的生成逻辑，但必须解决“来源引用可被后续步骤解析”的问题。

第一版推荐采用兼容策略：

1. 单文档模式继续输出 `PageN`，完全走旧逻辑。
2. 多文档模式在 Step1 前完成文档归并，但给 Step1 的上下文中明确标注来源。
3. 多文档 Step1 输出仍保持现有字段结构，同时在 `main_module_source/submodule_source/page` 中尽量保留可解析来源。
4. 多文档第一版不是“每个文档各自生成一套 Step1 后再简单相加”，而是先逐文档提取候选需求，再按系统维度合并成一套 Step1 输入。
5. 如果短期不改 Step2 / Step3 查询逻辑，则多文档第一版可以把辅助文档内容尽量吸收到模块描述和功能摘要中；如果要让 Step2 / Step3 继续精确回查所有来源文档，则只需要增加“来源解析适配层”，不需要重写 case/design 生成主逻辑。

结论：用户理解的方向是对的。第一版重构重点应放在 Step1 之前，让一个系统的所有文档先融合成现有 Step1 可以理解的数据；Step2 / Step3 暂时只做必要兼容，不主动重构生成逻辑。

## 0.1 多文档第一步到底如何生成和合并

多文档不是简单遍历文档后分别生成全部 Step1 数据，也不是把所有页面无脑拼成一个超长上下文。推荐逻辑是“三段式”：

```text
逐文档提取
  -> 跨文档归并
  -> 生成系统级 Step1 输入
```

### 逐文档提取

对每个文件分别读取 parse 页数据，并分别做页摘要。这个阶段只做“抽取”，不最终决定模块结构。

例如三个文档合计从 parse 表取到 30 页数据，流程应采用“核心文档集 + 补充文档增强”：

```text
30 页原始 parse 内容
  -> 30 页结构化页摘要
  -> 过滤不可测试页
  -> 从核心文档集生成候选功能基线
  -> 用补充文档增强已有候选项
  -> 仅在确实有独立可测试功能时少量新增
  -> 生成系统级 Step1 输入
```

不是把 30 页原文直接传给 Step1，也不是每个文档单独跑一套 Step1 再合并结果。

如果 A 是被系统自动识别出来的核心文档之一，A 单独可生成 10 条 Step1 数据，B/C 是补充文件，那么合理结果通常应是 10 条被增强后的数据，或者在 B/C 确实包含 A 未覆盖的独立可测试功能时增加到 11~12 条。绝不应该因为 A/B/C 一共 30 页，就膨胀成 30 条 Step1 数据。

每条页摘要至少包含：

```text
platform
document_id
document_name
document_type
page
source
requirement_type
section
title
description
business_rules
acceptance_criteria
source_type
testability
```

### 跨文档归并

归并目标是把多个文件中描述同一业务能力、同一页面、同一接口或同一规则的内容合成一个系统级候选需求。

归并依据：

1. `platform` 相同。
2. `section/title/description` 语义相近。
3. 同属一个业务域或功能域。
4. API 文档中的 endpoint 能对应到功能需求中的业务动作。
5. 业务规则文档能约束某个功能点。
6. 背景文档只补充术语、角色、流程和上下文，不单独生成模块，除非明确可测试。

归并后的优先级：

```text
需求输入/功能规格 > 接口设计 > 业务规则 > 报表/角色/权限说明 > 背景流程
```

合并行为：

1. 功能文档决定“是否生成模块”和“模块主名称”。
2. 业务规则补充到 `business_rules`、`acceptance_criteria` 或模块描述。
3. API/Interface 文档补充 endpoint、参数、状态码、接口约束。
4. Report/Role 这类文档补充报表、权限、角色检查点。
5. 背景 BPMN/Blueprint 补充流程上下文和依赖，不直接膨胀模块数量。
6. 如果多个文档描述冲突，不静默覆盖，先记录为冲突或在描述中标出待确认。
7. 补充文档默认执行 `enhance_existing`，只有满足“主需求未覆盖 + 明确可测试 + 独立功能边界”时才执行 `create_new`。
8. 无法归入主需求候选项的补充信息进入 `unmapped_supplement_list`，不直接生成 Step1。
9. 文档顺序不能作为主锚点判断依据，必须按内容评分、文档类型、可测试性和业务覆盖度自动识别。
10. 允许多个核心文档共同构成主锚点集，不强制全系统只有一个主文档。

### 生成系统级 Step1 输入

最终传给 Step1 的不是每个文件一份数据，而是系统级合并后的 Functional/API 列表。

示例：

```json
{
  "platform": "CSS-CL",
  "requirement_type": "Functional",
  "section": "Cash Management",
  "title": "GM FX Prompt maintenance",
  "description": "根据需求输入生成主功能描述，并合并背景流程、角色权限、接口和报表约束。",
  "business_rules": [
    "角色权限来自 CSS-CL User maint Role.xlsx",
    "接口约束来自 CSS-CL Interface Spec.doc"
  ],
  "acceptance_criteria": [
    "用户可按权限维护 Prompt",
    "接口请求和响应符合 Interface Spec"
  ],
  "page": "1,2,3",
  "source": "CSS-CL GM FX Prompt_Detail ver.docx; CSS-CL Interface Spec.doc; CSS-CL User maint Role.xlsx",
  "source_refs": "DOC2#Page1,DOC3#Page4,DOC5#Page2",
  "source_type": "web",
  "testability": true
}
```

这样 Step1 仍然看到的是“需求摘要列表”，但它已经包含多文档整合后的上下文。

## 0.2 实际项目示例

例如一个项目资料目录中有：

```text
1. 背景信息1/
  Cash Management BPMN with details.pdf
  Cash Management Service BluePrint1_v0.2.pdf
  CSS-CL Interface Spec.doc
  CSS-CL Report Spec.doc
  CSS-CL User maint Role.xlsx

2. 需求输入/
  CSS-CL GM FX Prompt_Detail ver.docx
  CSS-CL GM FX Prompt_Med ver.docx
  CSS-CL GM FX Prompt_short ver.docx

3. 输出样例/
  历史人工输出或参考样例
```

实际生成时，把“背景信息”和“需求输入”中的相关文件导入知识库，然后将这些文件 ID 一起传给 `kbSpaceFileId`。

生成目标仍然是当前三份数据：

```text
Test Module
Test Module Design
Test Case
```

其中“背景信息”主要提供业务流程、角色、接口、报表、权限和术语；“需求输入”作为主需求来源，决定主要模块、设计点和用例；“输出样例”默认不作为需求来源，除非明确要用它约束导出格式或覆盖口径。

## 0.3 参数、字段和表结构总览

### 0.3.1 新增入口参数

第一版不新增入口参数。

继续复用当前参数：

```text
kbSpaceFileId
source_type
pages
version
resumeMode
resumeSubmoduleName
fixedThreeLanguage
outputLanguage
```

`kbSpaceFileId` 的定义变化：

| 字段 | 旧定义 | 新定义 |
| --- | --- | --- |
| kbSpaceFileId | 单个知识库文件 ID | 单个或多个知识库文件 ID。多个 ID 可用逗号分隔或数组传入 |

### 0.3.2 新增内部变量

| 变量 | 定义 |
| --- | --- |
| kb_space_file_id_raw | 原始 `kbSpaceFileId` |
| file_id_list | 解析后的文件 ID 列表 |
| input_mode | `single_file` 或 `multi_file` |
| platform | 前置流程提供的系统名称 |
| document_manifest | 本次参与生成的文档清单 |
| page_summary_list | 多文档逐页摘要结果 |
| core_doc_set | 自动识别出的核心文档集合，可包含多个文档 |
| supplement_doc_set | 补充文档集合 |
| multi_doc_step1_input | 多文档合并后的 Step1 原始输入 |
| multi_doc_functional_list | 合并后的 Web/功能 Step1 输入 |
| multi_doc_api_list | 合并后的 API Step1 输入 |
| unmapped_supplement_list | 无法挂接到核心功能的补充内容，默认不生成 Step1 |

### 0.3.3 parse 表字段要求

| 字段 | 是否必需 | 定义 |
| --- | --- | --- |
| platform 或 system_name | 后续新增 | 系统名称，进入 Agent 后统一映射为 `platform` |
| document_id / documentId | 已有 | 文件 ID |
| fileName | 已有 | 文件名，用于来源追溯和文档角色识别 |
| page | 已有 | 文档内页码，不做跨文档唯一标识 |
| content / analysis_content | 已有 | 页面正文或解析正文 |

不要求 parse 表新增 `document_type`，文档角色由 Agent 内部临时识别。

### 0.3.4 三张核心表建议新增字段

优先复用现有字段，尽量少加字段。

| 表 | 最小新增字段 | 复用字段 |
| --- | --- |
| kb_gen_test_module | `evidence_summary` | `main_module_source`, `submodule_source`, `platform`, `documentId` |
| kb_gen_test_module_design | `evidence_summary` | `main_module_source`, `submodule_source`, `platform`, `documentId` |
| kb_gen_test_case | `evidence_summary` | `source_value`, `platform`, `document_id` |

如果后续发现必须精确按文件 ID 查询来源，再考虑新增 `source_document_ids`。第一版不强制新增。

### 0.3.5 现有字段定义变化

| 字段 | 旧定义 | 新定义 |
| --- | --- | --- |
| platform | Step1 AI 根据文档推断 | 前置流程提供的系统名称，Agent 内强制覆盖 |
| documentId / document_id | 单文档 ID | 兼容旧字段。多文档真实来源看 `source_document_ids` |
| page | 单文档页码 | 文档内页码，不保证跨文档唯一 |
| source / source_value / main_module_source / submodule_source | 多为 `PageN` | 单文档继续兼容 `PageN`；多文档改为 `fileName#PageN` 列表 |
| file_name | 单文件名 | 多文档时可使用 platform 或系统名作为导出基名 |

## 1. 背景

当前 `agt_gen_test_case_entry.txt` 的主流程以单个 `kbSpaceFileId` 作为输入，围绕单份需求文档完成：

1. 读取文档页内容。
2. 按页生成摘要。
3. 将摘要拆成 Background / Functional / Compliance / Other。
4. 生成 `combined_result` Markdown。
5. 将 Markdown 转为 Mermaid 思维导图。
6. 生成测试模块。
7. 生成测试设计点。
8. 生成测试用例。
9. 写入 KB 表并导出 xlsx。

实际项目中，需求资料通常由多份文档共同组成，例如背景说明、业务规则、功能需求、接口设计、原型说明、合规要求等。单文档流程会导致：

- 背景文档中的业务上下文无法被后续模块和用例稳定引用。
- 业务规则文档中的限制条件容易漏生成。
- 功能需求和接口设计分散在不同文档时，Web / API case 无法自然联动。
- 多文档页码会冲突，例如多个文档都有 Page1。
- 当前 `documentId + page` 的追溯模型无法表达一个 case 来自多个文档。
- 当前思维导图只作为展示结果，没有成为多文档联动的结构化中间产物。

本方案目标是将现有“单文档生成”升级为“项目级多文档联合生成”，同时保留单文档兼容能力。

## 2. 总体目标

重构后的流程应支持：

1. 输入单文件、多个文件或文件夹 ID。
2. 自动识别每份文档的类型和角色。
3. 逐文档、逐页提取结构化需求信息。
4. 将多文档内容归并为项目级需求原子。
5. 基于需求原子构建项目级需求图谱。
6. 从需求图谱生成 Mermaid 思维导图。
7. 后续模块、设计点、用例生成都基于需求图谱和上下文包。
8. 用例可追溯到多个文档和多个页面。
9. 支持项目级断点续跑。
10. 保持当前单文档输入的向后兼容。

## 3. 当前主流程分析

### 3.1 入口参数

当前入口主要参数：

```text
kbSpaceFileId
source_type
pages
version
resumeMode
resumeSubmoduleName
fixedThreeLanguage
outputLanguage
```

其中 `kbSpaceFileId` 是唯一文档入口，后续变量：

```text
documentId = kbSpaceFileId
effectiveDocumentId = kbSpaceFileId
```

这使整个流程天然绑定单文档。

### 3.2 文档读取

当前读取逻辑从 `kbs_doc_page_parse` 查询：

```sql
select page,
       case when analysis_content is null or analysis_content = ''
            then content
            else analysis_content
       end as content
from kbs_doc_page_parse
where documentId = "{{kbSpaceFileId}}"
order by cast(page as signed) asc
```

多文档后，这里需要从单个 `documentId` 查询变成多个文档的统一读取。

### 3.3 页摘要

当前对每页调用：

- `agt_page_content_summary`
- `agt_page_content_summary_api`

输出字段包括：

```json
{
  "page": 1,
  "requirement_type": "Functional",
  "section": "用户认证",
  "title": "登录功能需求",
  "description": "...",
  "business_rules": [],
  "acceptance_criteria": [],
  "dependencies": [],
  "testability": true
}
```

问题是缺少：

- `document_id`
- `document_name`
- `doc_type`
- `global_page_ref`
- `project_id`

因此不能可靠表达多文档来源。

### 3.4 Markdown 和思维导图

当前 `combined_result` 由四类 Markdown 拼接：

```text
# file_name
background_result
functional_result
compliance_result
other_result
```

然后通过 `tpl_txt_to_mermaid` 生成 Mermaid mindmap。

这个导图目前是文档摘要的可视化，后续 Step1 / Step2 / Step3 并不依赖它。多文档场景下，它应该升级为项目级需求图谱的可视化结果。

### 3.5 Step1 模块生成

当前 Step1 主要基于 Functional 摘要调用：

- `agt_test_doc_func_modules`
- `agt_test_doc_func_modules_api`

生成模块后写入 `kb_gen_test_module`。

当前核心字段包括：

```text
platform
main_module_name
main_module_source
submodule_name
submodule_description
submodule_source
documentId
version
source_type
page
```

多文档后 `main_module_source` 和 `submodule_source` 不能只使用 `PageN`，需要变为跨文档引用。

### 3.6 Step2 设计点生成

当前 Step2 根据模块中的 page/source 回查原始页面，然后调用：

- `agt_test_doc_design`

写入 `kb_gen_test_module_design`。

多文档后，Step2 需要支持跨多个 `documentId + page` 查询原文。

### 3.7 Step3 用例生成

当前 Step3 根据设计点中的 page/source 回查原文，然后调用：

- `agt_test_doc_gen_case`

写入 `kb_gen_test_case`。

多文档后，一个 case 的依据可能来自多份文档，因此 `source_value` 和 `document_id` 需要扩展。

## 4. 推荐目标架构

建议将流程拆成以下层级：

```text
输入层
  -> 文档清单层
  -> 页摘要层
  -> 需求原子层
  -> 需求图谱层
  -> 项目思维导图层
  -> 模块生成层
  -> 设计点生成层
  -> 用例生成层
  -> 导出与统计层
```

### 4.1 输入层

支持三种输入模式：

```text
single_file: kbSpaceFileId 中只有一个文件 ID
multi_file: kbSpaceFileId 中有多个文件 ID
folder_expanded: 外部或前置步骤把文件夹展开成多个文件 ID 后，仍传给 kbSpaceFileId
```

第一版不新增必填参数，复用当前参数：

```text
kbSpaceFileId
```

兼容原则：

- 如果 `kbSpaceFileId` 只包含一个 ID，走现有单文档逻辑。
- 如果 `kbSpaceFileId` 包含多个 ID，进入多文档预处理逻辑。
- 如果未来传入文件夹 ID，应先展开成文件 ID 列表，再复用 `kbSpaceFileId`。
- `project_id` 不作为入口参数，内部自动生成即可，例如取第一个文件 ID 或 `version + firstFileId`。

### 4.2 文档清单层

新增 `document_manifest`，用于描述项目下所有文档：

```json
{
  "project_id": "P-001",
  "documents": [
    {
      "document_id": "doc_background",
      "document_name": "背景说明.pdf",
      "doc_type": "background",
      "role": "context",
      "priority": 20,
      "alias": "BG"
    },
    {
      "document_id": "doc_business",
      "document_name": "业务需求.docx",
      "doc_type": "business",
      "role": "rule_source",
      "priority": 80,
      "alias": "BIZ"
    },
    {
      "document_id": "doc_functional",
      "document_name": "功能需求.docx",
      "doc_type": "functional",
      "role": "test_source",
      "priority": 100,
      "alias": "FUNC"
    },
    {
      "document_id": "doc_api",
      "document_name": "接口设计.yaml",
      "doc_type": "api_design",
      "role": "contract_source",
      "priority": 90,
      "alias": "API"
    }
  ]
}
```

文档类型建议：

```text
background
business
functional
api_design
ui_design
compliance
other
```

文档角色建议：

```text
context
rule_source
test_source
contract_source
constraint_source
reference
```

### 4.3 页摘要层

每页摘要必须携带文档身份：

```json
{
  "project_id": "P-001",
  "document_id": "doc_functional",
  "document_name": "功能需求.docx",
  "doc_type": "functional",
  "doc_role": "test_source",
  "doc_alias": "FUNC",
  "page": "3",
  "global_page_ref": "FUNC#Page3",
  "requirement_type": "Functional",
  "section": "用户认证",
  "title": "登录",
  "description": "...",
  "business_rules": [],
  "acceptance_criteria": [],
  "dependencies": [],
  "testability": true,
  "source_type": "web"
}
```

`global_page_ref` 是多文档流程的基础，不允许再只依赖裸 `PageN`。

### 4.4 需求原子层

新增需求原子，表示可被生成、归并、追溯的最小需求单元。

```json
{
  "atom_id": "REQ-FUNC-001",
  "project_id": "P-001",
  "title": "用户登录",
  "description": "用户可以使用账号密码登录系统。",
  "requirement_type": "Functional",
  "module_hint": "用户认证",
  "source_type": "web",
  "business_rules": [
    "连续登录失败 5 次后锁定账户"
  ],
  "acceptance_criteria": [
    "正确账号密码登录成功",
    "错误密码提示失败"
  ],
  "dependencies": [
    "REQ-BIZ-003"
  ],
  "evidence": [
    {
      "document_id": "doc_functional",
      "doc_type": "functional",
      "page": "3",
      "ref": "FUNC#Page3"
    },
    {
      "document_id": "doc_business",
      "doc_type": "business",
      "page": "8",
      "ref": "BIZ#Page8"
    }
  ]
}
```

需求原子生成规则：

1. 一条页摘要可以拆成多个需求原子。
2. 不可测试的背景内容不要直接生成 case，但可作为 context evidence。
3. 业务规则可独立成为原子，也可挂接到功能原子。
4. API endpoint 应成为独立原子，并与相关功能原子建立关系。
5. 每个原子必须有 evidence。

### 4.5 需求图谱层

需求图谱是多文档联动的核心，不建议只依赖 Mermaid 文本。

图谱节点：

```json
{
  "node_id": "M_AUTH",
  "node_type": "module",
  "name": "用户认证",
  "children": ["SM_LOGIN", "SM_LOGOUT"],
  "requirement_atom_ids": ["REQ-FUNC-001", "REQ-BIZ-003"],
  "evidence_refs": ["FUNC#Page3", "BIZ#Page8"]
}
```

节点类型：

```text
project
domain
module
submodule
requirement
business_rule
api_contract
acceptance_criteria
risk
conflict
```

图谱边：

```json
{
  "from": "REQ-FUNC-001",
  "to": "REQ-BIZ-003",
  "edge_type": "constrained_by"
}
```

边类型：

```text
contains
depends_on
constrained_by
implemented_by
verified_by
supplements
conflicts_with
```

### 4.6 项目思维导图层

Mermaid 思维导图应从 `requirement_graph` 渲染，而不是直接从 `combined_result` 反推。

推荐结构：

```text
Project
  Module
    Submodule
      Requirement
      Business Rule
      API Contract
      Acceptance Criteria
```

思维导图用途：

1. 展示多文档融合后的项目需求结构。
2. 辅助用户检查模块拆分是否合理。
3. 作为后续按模块续跑的可视化锚点。
4. 标识来源文档和冲突点。

限制：

1. Mermaid 不承载完整语义。
2. 后续生成不能只从 Mermaid 读取上下文。
3. Mermaid 必须能追溯到 `requirement_graph` 的节点 ID。

### 4.7 模块生成层

Step1 输入应从 `functional_list` 改为项目级 `module_context_packages`。

模块上下文包示例：

```json
{
  "project_id": "P-001",
  "module_hint": "用户认证",
  "primary_requirements": [],
  "business_rules": [],
  "api_contracts": [],
  "background_context": [],
  "acceptance_criteria": [],
  "evidence_refs": ["FUNC#Page3", "BIZ#Page8"]
}
```

Step1 输出建议：

```json
{
  "project_id": "P-001",
  "module_id": "M_AUTH",
  "submodule_id": "SM_LOGIN",
  "platform": "system name",
  "main_module_name": "用户认证",
  "main_module_source": "FUNC#Page3, BIZ#Page8",
  "submodule_name": "登录",
  "submodule_description": "...",
  "submodule_source": "FUNC#Page3, BIZ#Page8",
  "source_refs": ["FUNC#Page3", "BIZ#Page8"],
  "source_document_ids": ["doc_functional", "doc_business"],
  "requirement_atom_ids": ["REQ-FUNC-001", "REQ-BIZ-003"],
  "source_type": "web"
}
```

### 4.8 设计点生成层

Step2 输入应包含：

1. 当前模块和子模块。
2. 该子模块关联的需求原子。
3. 业务规则。
4. API 契约。
5. 背景上下文。
6. 验收标准。
7. 多文档页面证据。

Step2 不应再只通过 `PageN` 查询原文，而是通过 `source_refs` 跨文档查询。

### 4.9 用例生成层

Step3 输入应包含设计点完整上下文：

```json
{
  "project_id": "P-001",
  "module_id": "M_AUTH",
  "submodule_id": "SM_LOGIN",
  "design_id": "D_LOGIN_001",
  "design_point": {},
  "requirement_context": [],
  "business_rule_context": [],
  "api_context": [],
  "background_context": [],
  "evidence_pages": []
}
```

case 输出建议增加：

```text
project_id
module_id
submodule_id
design_id
source_refs
source_document_ids
requirement_atom_ids
```

`source_value` 可继续保留，但内容改为：

```text
FUNC#Page3; BIZ#Page8
```

## 6. 断点续跑设计

当前续跑主要依赖：

```text
version + documentId + submodule_name
```

多文档后建议改为：

```text
version + project_id + module_id/submodule_id
```

新增参数：

```text
resumeModuleId
resumeSubmoduleId
resumeDocumentId
```

兼容规则：

1. 如果传 `resumeSubmoduleId`，优先按 ID 续跑。
2. 如果只传 `resumeSubmoduleName`，走旧逻辑，但需要提示可能存在重名。
3. 如果传 `resumeDocumentId`，只重跑受该文档影响的需求原子、模块、设计点和 case。
4. `auto` 模式应按项目级最新进度判断，不再只看单文档。

## 7. 单文档兼容策略

单文档模式必须保持可用。

当只传 `kbSpaceFileId` 时：

1. 自动生成 `project_id = kbSpaceFileId` 或 `version + kbSpaceFileId`。
2. 自动构造一个 `document_manifest`。
3. `doc_alias` 可使用 `DOC`。
4. `global_page_ref` 为 `DOC#PageN`。
5. 旧字段 `documentId` 继续写入 `kbSpaceFileId`。
6. 新字段同步写入项目级信息。

这样可以逐步迁移，不需要一次性改完所有下游。

## 8. 生成策略

多文档结合时应遵循以下优先级：

```text
功能需求 / 接口设计 > 业务规则 > 合规要求 > 背景说明 > 其他参考
```

各类文档作用：

| 文档类型 | 作用 | 是否直接生成 case |
| --- | --- | --- |
| background | 补充业务背景、术语、上下文 | 通常否 |
| business | 补充业务规则、状态流转、限制条件 | 是 |
| functional | 主功能、主流程、界面行为 | 是 |
| api_design | endpoint、参数、状态码、schema | 是 |
| ui_design | 页面元素、交互、布局约束 | 是 |
| compliance | 合规、审计、安全要求 | 是 |
| other | 参考资料 | 视内容而定 |

## 9. 冲突处理策略

多文档中可能出现规则冲突，例如：

- 业务文档说密码错误 3 次锁定。
- 功能文档说密码错误 5 次锁定。

处理策略：

1. 不静默覆盖。
2. 写入 `kb_gen_requirement_conflict`。
3. 在图谱中生成 `conflict` 节点。
4. 生成 case 时优先使用高优先级文档。
5. 在导出结果中增加待确认清单。

冲突状态：

```text
open
auto_resolved
manual_resolved
ignored
```

## 10. 导出策略

当前导出路径基于单文件名：

```text
Test Module/test_module_${baseFileName}.xlsx
Test Module Design/test_module_design_${baseFileName}.xlsx
Test Case/test_case_${baseFileName}.xlsx
```

多文档后建议基于项目名：

```text
Test Module/test_module_${projectName}_${version}.xlsx
Test Module Design/test_module_design_${projectName}_${version}.xlsx
Test Case/test_case_${projectName}_${version}.xlsx
Requirement Graph/requirement_graph_${projectName}_${version}.xlsx
Requirement Conflict/requirement_conflict_${projectName}_${version}.xlsx
```

单文档兼容时继续使用原路径或保持现有命名。

## 11. 验收标准

重构完成后应满足：

1. 单文档输入结果与当前流程基本一致。
2. 多文档输入不会出现 Page1 冲突。
3. case 能展示多个来源，例如 `FUNC#Page3; BIZ#Page8`。
4. 思维导图展示项目级结构，而不是单文档摘要拼接。
5. 背景文档只补上下文，不制造大量无效 case。
6. 业务规则文档中的规则能进入设计点或 case。
7. API 文档能与功能需求建立关联，生成 API case。
8. 断点续跑可以按项目、模块、子模块恢复。
9. 导出文件包含多文档追溯信息。
10. 新增字段不破坏旧单文档调用。

## 12. 建议实施路线

### 阶段一：多文档输入和来源追溯

目标：

- 支持多文件输入。
- 页摘要增加文档身份。
- 全流程支持 `DOC#PageN`。

### 阶段二：需求原子

目标：

- 从页摘要生成需求原子。
- 初步支持相似需求合并。
- 建立 evidence_refs。

### 阶段三：需求图谱和思维导图

目标：

- 生成项目级 `requirement_graph`。
- Mermaid 从图谱渲染。
- Step1 基于模块上下文包生成。

### 阶段四：Step2 / Step3 上下文升级

目标：

- 设计点和 case 按 `source_refs` 跨文档查询。
- 引入业务规则、API 契约、背景上下文。

### 阶段五：续跑、导出、统计

目标：

- 项目级断点续跑。
- 项目级 xlsx 导出。
- 覆盖率和冲突报告。
