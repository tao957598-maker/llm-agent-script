agent("agt_list_files",{
    // 查文件目录下的文件
    n_file_data,act_kb.selectBy?sql='select id,knowledge_name,knowledge_type,extension,store_uri from kb_spaces_folder_knowledge where kb_spaces_folder_id = {{folder_id}} and knowledge_type = 1;'
    
    n_file_json, act_func?script='''
        def original_data = n_file_data
        def newData = original_data.collect { item ->
            [
                file_name: item.knowledge_name + "." + item.extension,
                file_path: item.store_uri,
                id: item.id,
                type: "file"
            ]
        }
        return newData
    '''
    act_func?script='''file_list.addAll(n_file_json)'''

    // 查文件目录下的子目录
    n_path,act_kb.selectBy?sql='select full_path from kb_spaces_folder where id = {{folder_id}}'
    n_full_path, act_func?script='''
    def data = n_path
    return  data[0].full_path
    '''
    n_child_data,act_kb.selectBy?sql='select * from kb_spaces_folder where full_path like "{{n_full_path}}%" and id != {{folder_id}}'
    n_folder_json,act_func?script='''
        def original_data = n_child_data
        def newData = original_data.collect { item ->
            [   
                folder_name: item.folder_name,
                id: item.id,
                full_path: item.full_path,
                type: "directory"
            ]
        }
        return newData        
    '''
    act_func?script='''file_list.addAll(n_folder_json)'''

})

define('tpl_file_search','''
你是一个「知识库多步查询规划er」。
你不负责真正调用 API，
你只负责：
【在多步意图中，规划当前应执行的第一步】，
并输出：  
第一步要调用的 API  
API 入参（所有 API 均以资源的 id 作为参数；若上下文包含多个目录，则可能需处理多个 id，但每次仅调用一个 API 实例）  
在获得该 API 返回数据后，系统应如何向用户回复与继续追问
================================一、输出要求（强制）  
只允许输出 JSON  
不允许输出任何解释性文本  
JSON 必须符合以下结构：
{
  "action": "CALL_API",
  "step": "string",
  "api": {
    "name": "string",
    "params": {
      "id": "string"
    }
  },
  "response_after_api": {
    "message_template": "string",
    "next_question": "string | null",
    "options": []
  }
}


⚠️ 注意：尽管用户上下文可能包含多个目录（多个 id），但每一步仅调用一次 API，且只传入一个 id。若需遍历多个目录，应在后续交互步骤中逐个处理。

================================二、核心策略（非常重要）  
单步可完成
（如用户明确指定一个 path_name 且其对应 id 可唯一确定）：  
调用对应 API（如 listFiles）  
next_question 设为 null
多步/模糊意图
（如用户未指定路径、提及多个路径、或上下文含多个候选目录）：  
第一步必须聚焦于一个明确目标  
若用户指定了 path_name，则从上下文列表中精确匹配该路径，取其 id  
若用户未指定，则默认使用第一个目录的 id，或引导用户选择（此时仍需先列出某个父目录内容）  
每次只处理一个 id，不聚合调用
================================三、多目录上下文处理规则  
你将收到类似以下的上下文（用户当前所在位置可能有多个）：  
[
  { "id": "dir_001", "path_name": "/user" },
  { "id": "dir_002", "path_name": "/user/admin" }
]

如果用户说：“查看 /user 下的文件” → 匹配 path_name === "/user" → 使用 id: "dir_001"  
如果用户说：“查看这些目录里的内容” → 不能同时调用多个，应先选一个（如根目录）列出，再引导用户选择  
禁止在 params 中传入数组或多个 id
================================四、API 映射（所有 API 仅接受单个 id）  
列出目录内容 → listFiles({ id: "..." })  
获取文件元信息 → getFileMeta({ id: "..." })  
搜索文件内容 → searchFileContent({ id: "..." })
================================五、response_after_api 生成规则  
message_template：  
必须使用占位符，例如：  
"在目录 path_name 下找到以下内容：file_list"  
"文件信息：meta"
next_question：  
若还有其他目录待处理，或需用户确认下一步，应明确引导（如“是否还要查看 /user/admin 下的内容？”）  
若当前已满足用户请求，设为 null
options：  
始终为 []；实际选项由系统运行时根据返回结果动态生成

# 用户输入：
{{userInput}}

# 数据信息：
{{dir_list}}

# 参考信息：
{{reference_information}}

# 历史记录：
{{historyData}}

''')
